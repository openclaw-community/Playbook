AI-Assisted Software Development Playbook

From concept to shipped product using AI as architect and builder — a repeatable framework with a real-world reference implementation.

Matthew Grunert
Reference: OpenClaw Hub Dashboard (v1.1.0)
February 2026
📎
Reference implementation — how it played out in the OpenClaw Hub project
⚡
After-the-fact learning — what we'd do differently
0
Pre-Conditions & Environment Setup
0.1 — Establish a Secure, Isolated Environment
  • AI agents with code execution access should operate in a controlled environment
  • Network isolation, restricted accounts, and clear boundaries between experimentation and production
  • Define what the AI can and cannot touch (repos, APIs, credentials, infrastructure)
📎
Mac Mini M4 on an isolated VLAN behind double NAT. Dedicated limited macOS user account. OpenClaw runs headless with Telegram integration. No access to production systems.
0.2 — Choose Your Model Roles
  • Architect model: Higher-capability model for design, planning, spec writing, review, and decision support
  • Builder model: Capable execution model for code generation, testing, and commits against a real codebase
  • The architect should never touch the codebase directly. The builder should never freelance on design decisions.
  • These can be the same model in different contexts, or different models entirely
  • Cost consideration: If your builder agent consumes API tokens, do the heavy iteration, wireframing, and design exploration with a web-based LLM (e.g., claude.ai, ChatGPT) where usage is covered by a subscription rather than per-token billing. Save the API-token-consuming agent for execution. The design phase involves extensive back-and-forth that would be expensive at API rates.
  • Some of this multi-role workflow can also be accomplished with sub-agents or agent teams in tools like Claude Code, but for non-developers, the manual separation between a web-based architect and a local builder agent is a practical and accessible starting point
📎
Opus 4.6 (via claude.ai, subscription) = Architect. Sonnet 4.6 (via OpenClaw locally, API tokens) = Builder. All the iterative design work — prototyping, UX feedback loops, spec writing — happened in the web session at no per-token cost. Sonnet only consumed API tokens during execution, where the work was focused and efficient.
⚡
The two-model separation wasn't just cost optimization — it enforced discipline. The architect couldn't take shortcuts because it didn't have repo access. The builder couldn't drift from the plan because it had a written spec to follow.
0.3 — Ensure Version Control Is Ready
  • Clean repo, clear branch strategy, and a defined merge workflow before any AI writes code
  • The AI should be creating feature branches, not committing to main
📎
GitHub repo (openclaw-community/openclaw-hub) with a feature/dashboard branch. All 5 phases committed to the feature branch, merged to main only after full regression.
1
Concept & Requirements Definition
1.1 — Describe What You Want in Plain Language
  • Start with the problem, not the solution
  • What does the user need? What data exists? What's missing?
  • Don't constrain yourself to what you think is technically feasible — let the architect assess that
📎
"I want a dashboard that shows configured connections, token usage by day/week/month, and lets me estimate cost by setting a cost-per-token value that gets stored locally." — That was the entire starting prompt.
1.2 — Architect Assesses Feasibility and Expands Scope
  • The architect reviews existing codebase, data models, and infrastructure
  • Identifies what already exists that can be leveraged (e.g., existing logging, schemas)
  • Proposes additions or modifications to the original concept
  • Flags risks, dependencies, and architectural decisions that need human input
📎
Opus reviewed the GitHub repo, identified that SQLite was already logging token counts, cost, latency, and provider info on every request. Proposed expanding the concept to include connection CRUD management, encrypted credential storage, budget alerts, and a template-driven service type system — well beyond the original ask.
⚡
The product owner drove the vision — the architect helped realize it. The human identified the need for a full management UI, a proper landing page, and the ability to import existing connections. The architect's value was in proposing the mechanisms: template-driven Add Connection forms, Fernet encryption, the specific Import from .env implementation pattern.
1.3 — Make Design Decisions Early
  • Prompt the architect to ask clarifying questions about your requirements — and keep prompting until it has no more questions
  • Not all models will proactively surface ambiguities; you may need to explicitly ask "What questions do you have?" or "What assumptions are you making?"
  • Answer these now, not during implementation — every deferred decision becomes a blocker or a guess later
  • Common early decisions: auth model, data storage approach, UI framework (or lack thereof), scope boundaries
📎
Key early decisions: single-file HTML dashboard (no React, no npm, no build step), Chart.js from CDN, FastAPI serves everything, JSON file storage for config alongside SQLite, Fernet encryption for credentials, auth gate deferred to a future PR.
⚡
The "single-file HTML" decision was one of the best calls. It eliminated an entire class of frontend toolchain complexity and kept the dashboard shippable by a model that didn't need to manage node_modules.
2
UX Prototyping & Visual Design
2.1 — Build a Working Prototype Before Any Backend Code
  • The architect creates an interactive prototype (React, HTML, or whatever renders quickly)
  • The purpose is to see and interact with the design — click through pages, check form flows, validate layout
  • This is not throwaway work — it becomes the visual specification for the builder
📎
Opus built a full React prototype with simulated data — 4 pages (Overview, Connections, Costs, Activity), working navigation, Add Connection modal that adapted fields by service type, Chart.js graphs with mock data. This was reviewed and iterated on before any spec was written.
2.2 — Iterate on UX With the Architect
  • Review the prototype as a user, not a developer
  • Ask: Does this flow make sense? Is anything confusing? What's missing?
  • Push back on defaults — the architect will make reasonable assumptions, but your domain knowledge matters
📎
Multiple rounds of UX feedback: how the Add Connection form should adapt based on service type (LLM vs. GitHub token vs. media API), what fields each type needs, how cost-per-token config should be structured, what the Overview dashboard should prioritize.
⚡
Time spent on the prototype saved massive time during implementation. Sonnet had zero ambiguity about what the dashboard should look like because a pixel-accurate reference existed.
2.3 — Finalize the Prototype as Design Documentation
  • Save the prototype file in the repo (e.g., docs/dashboard-prototype.jsx)
  • This becomes living design documentation for future contributors
  • The builder will reference this for every visual decision
📎
Prototype saved to docs/dashboard-prototype.jsx in the repo. The implementation spec explicitly references it: "The rendered prototype IS the specification — every visual detail is intentional and should be replicated."
3
Specification Writing
3.1 — The Architect Produces a Comprehensive Implementation Spec
  • This is the most critical artifact in the entire workflow
  • The spec must be detailed enough that a different model (or person) can execute against it with zero ambiguity
  • It should cover: database schema, API endpoints with request/response examples, file structure, CSS values, testing commands, documentation updates
  • Think of it as a contract between the architect and the builder
📎
1,650-line spec covering 15 sections: project overview, existing architecture, phased implementation plan, database schema, 18 API endpoints with full JSON examples, static file serving, connection CRUD flows, cost configuration, complete visual spec (every hex color, font size, border radius, spacing value, Chart.js config), service template reference table, security model, testing checklist with curl commands, documentation update instructions, and future work.
⚡
The spec's level of detail felt excessive while writing it. It wasn't. Every section that was thorough shipped clean. Every section where we hand-waved ("Sonnet will figure it out") required back-and-forth during implementation.
3.2 — Include a Phased Implementation Plan
  • Break the work into independently testable phases
  • Each phase should produce something that works and can be verified before the next phase starts
  • Define branch naming, commit message conventions, and merge strategy
📎
Five phases: (1) DB schema + data layer, (2) API endpoints, (3) HTML dashboard + static serving, (4) Connection CRUD, (5) Cost config + budget alerts. Each phase had its own acceptance criteria and testing checklist.
3.3 — Include Explicit Testing Criteria
  • Curl commands, expected outputs, browser verification steps
  • If you can't test it, you can't verify it, and the builder will assume it works
📎
Section 12 of the spec: 29 test cases with exact curl commands and expected JSON responses. The final regression test confirmed 29/29 pass.
3A
Repository Structure & Documentation Infrastructure
3A.1 — Establish Community-Standard Repo Files
  • Before any feature code is written, the repo should have the foundational files that signal a well-maintained project
  • These files also give the builder agent critical context about how the project operates
  • Standard files at the repo root: README.md, LICENSE, CONTRIBUTING.md, CODE_OF_CONDUCT.md, SECURITY.md, CHANGELOG.md, .env.example, .gitignore
📎
OpenClaw Hub had all of these in place before the dashboard work began. When Sonnet cloned the repo, it had immediate context on coding standards, contribution workflow, and project conventions without needing to ask.
⚡
GitHub's template system for CODE_OF_CONDUCT.md is not where most guides say it is. The actual workflow: create a new file, name it CODE_OF_CONDUCT.md, and GitHub auto-offers the Contributor Covenant template. Settings → Community Standards no longer has the "Add" button that older tutorials reference.
3A.2 — Create Project Tracking Documents
  • These are the "external brain" for AI-assisted development — they maintain continuity across sessions and models
  • Keep these in a docs/ folder or a separate docs repo
  • STATUS.md — Current version, what's done, what's in progress, what's next
  • ARCHITECTURE.md — System design, module relationships, data flow
  • CURRENT_TASK.md — The single most important file for AI continuity. This is the first thing any LLM should read when resuming work.
📎
OpenClaw Hub used CURRENT_TASK.md as the continuity handoff between sessions. Each session ended with an update to this file. When OpenClaw resumed work the next day, the first instruction was always: "Read CURRENT_TASK.md."
⚡
CURRENT_TASK.md is the single highest-value document in the entire system for non-developers working with AI agents. Without it, every new session starts from scratch. With it, the agent picks up exactly where it left off. Treat updating this file as part of the definition of "done" for every session.
3A.3 — Set Up Branch Protection and Workflow Rules
  • Protect the main branch before any AI agent starts committing code
  • Require pull requests for merges to main (even if you're the only reviewer)
  • Define a branch naming convention the builder will follow (e.g., feature/12-dashboard, bugfix/15-credential-leak)
  • Consider creating a team or role for bot accounts if your builder agent pushes directly
📎
Branch protection enabled on main: required PR reviews, status checks when CI is configured. Sage's bot account (sageopenclawbot) was added to an automation team with bypass permissions so it could push to feature branches without being blocked.
⚡
If you're using GitHub rulesets (the newer system), note that bypass lists only show roles, teams, and apps — not individual users. Create a team (e.g., automation) and add your bot account to it.
3A.4 — Structure the Builder's Handoff Documents
  • When the architect produces a spec, it's not the only document the builder needs
  • The builder also needs structured instructions on how to work with the codebase
  • Create a handoff prompt that includes: which files to read first, instruction to inspect actual codebase before implementing, phase-by-phase execution order, how to handle divergences, commit message conventions, how to update tracking documents
📎
The handoff prompt specified: read the full spec, clone the repo, inspect main.py, models.py, database.py, manager.py, and requirements.txt before writing any code. Implement Phase 1 first, test, then proceed. Do not skip ahead.
⚡
The handoff prompt is an underrated artifact. A well-written spec with a poor handoff produces mediocre results because the builder doesn't know how to approach the work. Think of it as onboarding a new contractor — you wouldn't just hand them a requirements doc and walk away.
3A.5 — Preserve Design Artifacts in the Repo
  • Prototypes, specs, and design documents belong in the repo — not just in chat history
  • These serve as design documentation for future contributors and future AI sessions
  • Suggested locations: docs/DASHBOARD-SPEC.md, docs/dashboard-prototype.jsx, docs/ARCHITECTURE.md
📎
Both DASHBOARD-SPEC.md (1,650 lines) and dashboard-prototype.jsx were committed to the repo's docs/ folder.
⚡
Chat history is ephemeral — repos are permanent. If a design decision only lives in a conversation, it will be lost when context windows roll over or conversations are archived. Anything that influenced the architecture belongs in the repo.
3A.6 — Define Documentation Updates as Part of Every Release
  • Documentation updates are not optional follow-up work — they're part of the release
  • Every merge to main should include updates to: README, CHANGELOG, STATUS.md, .env.example
  • Include this requirement in the spec so the builder does it automatically
📎
Section 13 of the dashboard spec explicitly listed every file to update with the exact content to add. The v1.1.0 merge included: README with dashboard section, CHANGELOG entry, STATUS.md version bump, .env.example with DASHBOARD_SECRET_KEY. None of this was left as a follow-up task.
4
Handoff to the Builder
4.1 — Prepare the Handoff Prompt
  • Give the builder both files: the spec and the prototype
  • Include explicit instructions: read the full spec first, inspect the actual codebase before coding, implement phase by phase, do not skip ahead
  • Tell the builder to flag divergences between the spec and reality rather than guessing
📎
Handoff prompt included: "Start by reading the full spec. Then clone the repo and inspect the existing codebase — particularly [specific files listed]. The spec was written based on the README; actual column names and patterns may differ slightly, so adapt accordingly. Implement Phase 1 first, test it, then proceed."
⚡
The instruction to "inspect the real code before implementing" was crucial. Sonnet immediately found 5 places the spec's assumptions didn't match reality (column names, ORM patterns, status field types, async patterns, route conflicts). Without this instruction, it would have built against the spec and produced code that didn't integrate.
4.2 — Create GitHub Issues Before Writing Code
  • Have the builder create issues for each phase before any implementation starts
  • Review the issues to confirm the builder understood the spec correctly
  • This catches misunderstandings before they become code
📎
5 GitHub issues (#21–#25) created by Sonnet, reviewed by us, before any code was written. Each issue included scope, acceptance criteria, and dependencies.
4.3 — Let the Builder Flag Problems
  • Expect divergences between the spec and the actual codebase
  • The builder should report these, not silently work around them
  • Bring divergences back to the architect for a decision
📎
Sonnet flagged: database uses timestamp not created_at, status is boolean not string, ORM is fully async (spec assumed sync), root endpoint already occupied, provider manager pattern different from spec assumption. Each was resolved before coding continued.
⚡
This is where the two-model pattern really pays off. The architect designs against documentation; the builder discovers reality. The human decides how to reconcile them. This mimics how senior architects and junior developers work together.
5
Implementation & Iteration
5.1 — Build Phase by Phase
  • The builder implements one phase at a time
  • Each phase is committed, tested against acceptance criteria, and confirmed before the next starts
  • Resist the temptation to let the builder "just keep going" — verification at each stage prevents compounding errors
📎
Phase-by-phase commits: 12076f4 (schema), 26f05f9 (API), bf... (HTML dashboard), bf8eb4d (CRUD), 8898b19 (cost config + extras). Each tested independently.
5.2 — Handle Emergent Requirements
  • During implementation, gaps will surface that nobody anticipated
  • These are not failures of planning — they're natural consequences of building against real code
  • Decide in real time: fix now (if small) or file as a follow-up issue (if large)
📎
"Import from .env" button — not in the original spec. When the Connections page was empty because existing providers were in .env but not the new database, we designed and built the import feature on the spot.
5.3 — Run Full Regression Before Merge
  • Test everything from every phase, not just the latest changes
  • Verify that existing functionality (health checks, API routes, model listing) is unaffected
  • Only merge to main after full pass
📎
29/29 regression tests passed. Root /, /health, /v1/models all unaffected. Credential masking confirmed. Import-from-env idempotent. Merge commit 4d65fd3, tagged v1.1.0.
6
Post-Launch Review & Iteration
6.1 — Use the Product Yourself
  • The best bug reports come from actually using what was built
  • Take notes on what feels wrong, what's missing, what's confusing
  • Don't rationalize issues away — if something feels off, it is off
📎
Morning after shipping, hands-on review found: (1) budget limits were global-only, not per-connection, (2) a phantom gpt-4o-mini test entry couldn't be deleted, (3) no way to add new cost entries, (4) configured connections didn't auto-populate cost entries.
⚡
These were all things that technically "worked" but didn't match how a real user would expect the product to behave. Automated tests passed; human judgment caught the gaps.
6.2 — Categorize Issues: Bugs vs. Enhancements vs. Architecture Gaps
  • Bugs: Something doesn't work as specified (orphaned test data, missing delete button)
  • Enhancements: Something works but could be better (chart formatting, historical navigation)
  • Architecture gaps: A design assumption was wrong and needs rethinking (global vs. per-connection budgets)
  • Treat each category differently — bugs are quick fixes, enhancements are scoped PRs, architecture gaps need architect involvement
📎
Bugs: phantom test entry, no delete button. Enhancements: daily breakdown in week/month charts, historical navigation arrows, stacked area charts for monthly view. Architecture gaps: per-connection budgets replacing global, separate api_calls table for non-LLM traffic.
6.3 — Draft Issues With the Architect, Execute With the Builder
  • The same workflow applies to iteration as to the initial build
  • Architect writes detailed issues with acceptance criteria
  • Builder executes against them
  • Don't shortcut this — "just fix it" instructions produce "just kind of works" code
📎
Three separate issue documents drafted by Opus: cost configuration overhaul, chart formatting improvements, and non-LLM traffic instrumentation. Each had full acceptance criteria and implementation guidance.
⚡
The iteration issues were more detailed than the original spec's individual phases, because we now had concrete examples of what "wrong" looked like. Specificity increased with experience.
6.4 — Track the Backlog and Prioritize
  • Not everything needs to be fixed immediately
  • Prioritize based on: user impact, dependency order, and what unblocks other work
  • Some issues are better deferred (auth gate, mobile responsive, dark/light toggle)
📎
Priority order: (1) cost config overhaul (broken UX), (2) non-LLM traffic instrumentation (blocks realistic demo data), (3) chart improvements (visual polish), (4) landing page (first impressions), (5) npm installer (distribution). Deferred: auth gate, push notifications, self-healing, mobile layout.
7
Documentation & Knowledge Capture
7.1 — Update All Project Documentation
  • README, CHANGELOG, STATUS, .env.example — all updated with each release
  • Don't leave this to "later" — it's part of the definition of done
📎
v1.1.0 documentation updates: README with dashboard section and service template table, CHANGELOG entry, STATUS.md version bump, .env.example with DASHBOARD_SECRET_KEY.
7.2 — Preserve the Spec and Prototype in the Repo
  • The implementation spec and prototype are valuable artifacts for future contributors
  • They explain not just what was built, but why and how
📎
docs/DASHBOARD-SPEC.md and docs/dashboard-prototype.jsx committed to the repo.
7.3 — Capture the Process, Not Just the Product
  • This playbook itself is a deliverable
  • Each project cycle should refine it with new learnings
  • The goal is a repeatable, improvable process — not a one-off success story
⚡
The biggest risk with AI-assisted development isn't that it doesn't work — it's that you can't explain how it worked, reproduce it, or teach it to someone else. Documentation of the process is as important as documentation of the code.

Key Principles

1
You are the product owner. AI doesn't decide what to build. You do.
2
Separate planning from execution. The architect designs; the builder builds. Mixing the roles produces mediocre results in both.
3
Spec before code. The spec is a contract. Ambiguity in the spec becomes bugs in the code.
4
Prototype before spec. Seeing the design prevents an entire category of misunderstandings.
5
Inspect reality before implementing. The codebase always diverges from documentation. The builder must read the actual code.
6
Phase and verify. Ship in testable increments. Verify each one before proceeding.
7
Flag, don't guess. When the builder encounters something unexpected, it reports back — it doesn't improvise.
8
Use what you build. The best QA is putting yourself in the user's seat.
9
Iterate with the same rigor as the initial build. Post-launch fixes deserve the same spec → issue → build → test workflow.
10
Document the process, not just the product. If you can't teach it, you can't scale it.
